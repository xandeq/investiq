# v1.5 Implementation Context — AI Portfolio Advisor

**Phase Goal:** Deliver Portfolio Health Check, AI-driven recommendations, Smart Screener personalization, and Entry Signals in a single unified advisor experience (`/advisor` page).

**4 Core Modules:**
1. Portfolio Health Check (ADVI-01) — diagnostic
2. AI Advisor (ADVI-02) — personalized recommendations
3. Smart Screener (ADVI-03) — complementary assets
4. Entry Signals (ADVI-04) — buy signals with fundamentals context

---

## Decision 1: Smart Screener Personalization Strategy

**LOCKED DECISION:** Smart Screener shows **complementary assets** (Option A)

**Rationale:**
- Diversification as core differentiator — if user wants "more of same," they already know what to buy
- Feeds directly from `compute_portfolio_health()` which identifies sector gaps, concentration risks, and missing asset classes
- Averaging down is secondary use case → can become a filter toggle later

**Code Context:**
- `backend/app/modules/advisor/service.py:compute_portfolio_health()` identifies:
  - `sector_map` — which sectors are concentrated
  - `biggest_sector_pct`, `biggest_asset_pct` — concentration risks
  - `underperformers` — weak positions to trim or complement
- Screener universe exists: `/acoes/screener` (900 tickers), `/renda-fixa` (Tesouro/CDB/LCI/LCA)
- Frontend: `frontend/src/features/acoes_screener/components/AcoesUniverseContent.tsx`, `RendaFixaContent.tsx`

**Implementation Implication:**
- Smart Screener filter = "Filter by sectors NOT in portfolio" + "Filter by missing asset classes"
- Rank by: alignment with portfolio gaps, sector diversity score, existing fundamentals
- Entry point: Health Check shows "Biggest Risk" + suggests "Complementary Assets to Address This"

## Decision 2: Health Check → Recommendations Flow (Page Architecture)

**LOCKED DECISION:** **Diagnostic-first flow** (Option A)

**Rationale:**
- Trust > speed — user needs to understand WHY they're receiving a recommendation before acting
- Health Check (score + biggest risk + underperformers) → contextualized recommendations creates confidence
- Target ICP (conservative/renda investor) values transparency over action bias

**Code Context:**
- `PortfolioHealth` schema (backend/app/modules/advisor/schemas.py):
  ```
  health_score: int (10-100)
  biggest_risk: str (1-sentence diagnosis)
  passive_income_monthly_brl: Decimal
  underperformers: list[str]
  data_as_of: datetime
  ```
- `run_portfolio_advisor()` (backend/app/modules/ai/skills/portfolio_advisor.py) returns:
  ```
  diagnostico, pontos_positivos, pontos_de_atencao, sugestoes, proximos_passos
  ```
- Frontend: `AdvisorContent` component (to be created/updated)

**Implementation Implication:**
- `/advisor` page structure:
  1. **Health Check Card** (top) — score + biggest_risk + passive_income + underperformers
  2. **AI Diagnosis Section** — portfolio_advisor narrative (diagnostico + pontos_positivos/atencao)
  3. **Actionable Recommendations** — Smart Screener + Entry Signals below the fold
- User reads diagnosis first, then sees recommendations contextualized to their specific situation

## Decision 3: Entry Signals Scope & Context (Hybrid Approach)

**LOCKED DECISION:** **Hybrid approach** (Option C)

**Details:**
- **Owned Assets (Portfolio + Watchlist):** Signals calculated on-demand/near-realtime
- **Universe Recommendations:** Signals calculated in daily batch (Celery beat)
- Rationale: Balances API cost (no spiking universe calculations) with portfolio monitoring (users get current signals for what they own)

**Code Context:**
- Entry Signals foundation already exists:
  - `backend/app/modules/swing_trade/service.py` — RSI + MA calculation
  - `backend/app/modules/opportunity_detector/agents/recommendation.py` — generates buy suggestions
  - Schema: `suggested_amount_brl`, `target_upside_pct`, `timeframe_days`, `stop_loss_pct`
- Screener snapshots (900 tickers) updated nightly by Celery beat

**Implementation Implication:**
- On-demand signals job for user's positions: triggered when page loads, cached <5min
- Batch signals job (daily): runs on Celery beat, populates universe recommendations
- Frontend shows both but with different freshness indicators ("Updated now" vs "Daily")
- API routes:
  - `GET /advisor/signals/portfolio` — on-demand, user's owned assets
  - `GET /advisor/signals/universe` — cached daily from Celery beat

## Decision 4: Portfolio Health Refresh Cadence

**LOCKED DECISION:** **Manual refresh + auto-invalidation on transaction** (Option C)

**Details:**
- Default: Show cached health (hourly or on-load)
- Refresh trigger: User clicks "Atualizar" button OR imports positions/adds transaction
- Rationale: On-demand every load is wasteful (deterministic SQL <200ms but Celery jobs can spike); cache hourly creates staleness confusion on imports; manual + trigger is what users expect from banking UX

**Code Context:**
- `compute_portfolio_health()` is deterministic, no external calls, <200ms
- Dashboard digest tasks already batch-calculate overnight (backend/app/modules/dashboard/digest_tasks.py)
- Transaction import pipeline exists: import/add position → can trigger cache invalidation

**Implementation Implication:**
- Cache key: `portfolio_health:{tenant_id}` with TTL=3600 (1 hour)
- Manual refresh endpoint: `POST /advisor/health/refresh` — bypasses cache, returns fresh
- Cache invalidation triggers:
  - `POST /transactions` (add transaction)
  - `POST /imports` (CSV/PDF import completion)
  - User clicks refresh button
- Frontend: Show "Last updated: X minutes ago" + "Atualizar" button
