Create `backend/tests/test_phase14_quality.py` with these test categories:

**1. Narrative quality validation (AI-05)** -- Tests that validate LLM narrative output does not hallucinate:

Use the DCF task's prompt construction logic -- the prompt includes the ticker, so the response should reference it.
- `test_narrative_no_hallucinated_metrics`: Create a mock narrative. Verify it does NOT contain financial values that were not in the input data (e.g., if fair_value is R$25.00, the narrative should not mention R$50.00).
- `test_narrative_fallback_on_llm_failure`: Verify that when `call_analysis_llm` raises `AIProviderError`, the static fallback narrative is used (not None, not empty).
- `test_narrative_language_pt_br`: Verify the LLM prompt includes "portugues" or "PT-BR" directive (inspect prompt construction in tasks.py).
- `test_narrative_length_bounds`: A valid narrative should be between 50 and 500 characters (not too short to be useless, not too long to overwhelm).

**2. Sensitivity analysis validation (AI-06)** -- Tests using `calculate_dcf_with_sensitivity` directly:
- `test_sensitivity_bear_less_than_base_less_than_bull`: Call with realistic inputs (fcf=1e9, shares=1e9, growth=0.08, wacc=0.12, terminal_growth=0.03). Assert scenarios["bear"]["fair_value"] < scenarios["base"]["fair_value"] < scenarios["bull"]["fair_value"].
- `test_sensitivity_range_coherence`: fair_value_range["low"] should equal bear scenario fair_value, fair_value_range["high"] should equal bull scenario fair_value.
- `test_sensitivity_with_10_sample_inputs`: Parameterize with 10 different (fcf, growth, wacc) combinations covering edge cases (high growth, low growth, high wacc, negative net_debt). ALL must satisfy bear < base < bull ordering.
- `test_sensitivity_key_drivers_present`: Result must include key_drivers list with at least 1 entry.
- `test_sensitivity_projected_fcfs_increasing`: With positive growth, projected FCFs should be monotonically increasing over the 5-year projection.

**3. Custom assumptions validation (AI-07)** -- Tests proving DCF responds to custom inputs:
- `test_custom_growth_rate_changes_output`: Call `calculate_dcf_with_sensitivity` with growth=0.05, then growth=0.15. Higher growth must produce higher fair_value.
- `test_custom_discount_rate_changes_output`: Call with wacc=0.08, then wacc=0.20. Higher discount rate must produce lower fair_value.
- `test_custom_terminal_growth_changes_output`: Call with terminal_growth=0.02, then terminal_growth=0.04. Higher terminal growth must produce higher fair_value.
- `test_assumptions_proportional_response`: Double the growth rate. Fair value should increase (not necessarily double, but measurably increase).
- `test_dcf_request_schema_validates_ranges`: Verify DCFRequest rejects growth_rate > 0.20, discount_rate > 0.30, terminal_growth > 0.05 (Pydantic validation).

**4. Cost tracking validation:**
- `test_cost_log_created_on_analysis`: Mock the DB session and verify `log_analysis_cost()` is called with correct analysis_type and status.
- `test_cost_estimate_known_provider`: `estimate_llm_cost("openrouter", "openai/gpt-4o-mini", 100, 50)` returns a positive float.
- `test_cost_estimate_unknown_provider`: `estimate_llm_cost("unknown", "model", 100, 50)` returns 0.0.
- `test_cost_estimate_free_tier`: `estimate_llm_cost("groq", "llama-3.3-70b-versatile", 1000, 500)` returns 0.0.

Use `pytest.mark.parametrize` for the 10-sample sensitivity test. Import directly from the modules -- no HTTP calls needed for calculation tests.
