# InvestIQ Pre-v1.8 Audit — 2026-04-19

## Executive Summary

**Five findings ranked by severity:**

**1. CORS blocks Idempotency-Key header (P0 — Security/Functional).** The billing duplicate-subscription hotfix (migration 0029, UpgradeCTA idempotency key) is silently broken in production. `app/main.py:70` allows only `["Content-Type", "Authorization", "X-Requested-With"]`. The `Idempotency-Key` header sent by the frontend `createCheckoutSession()` call is stripped by the browser CORS preflight when the frontend (`investiq.com.br`) calls the backend (`api.investiq.com.br`) — a cross-origin pair. The backend receives no key and never checks the `idempotent_checkout_requests` table. Fix: add `"Idempotency-Key"` to `allow_headers` in `app/main.py`.

**2. Macro rates (CDI/SELIC/IPCA) are null in production (P0 — Data Pipeline).** `GET /screener/macro-rates` returns `{"cdi": null, "ipca": null}` and the `/health/freshness` endpoint confirms `macro_stale: true, macro_last_fetched: null`. This means the `refresh_macro` Celery beat task has never run or Redis was flushed without recovery. The v1.6 Comparador feature and the v1.7 Simulador both depend critically on CDI/IPCA/SELIC. Additionally, the deployed backend response is missing the `selic` field entirely, indicating the backend was not re-deployed after feat(27-01) added selic support. Fix: verify Celery beat is running on VPS and force-run `refresh_macro` task; re-deploy backend.

**3. 18 test failures out of 645 (P1 — Tech Debt).** Full test run: 620 passed, 18 failed, 7 skipped. Key failure clusters: (a) dashboard `asset_allocation` test fails because `portfolio_daily_value` has no ORM model — the table is created only by migration, so the SQLite test engine never creates it; (b) `test_brapi_client_fetch_fundamentals_falls_back` fails because the function now returns a `variacao_12m` key but the test asserts exact equality with a 4-key dict; (c) `test_email_content_contains_key_info` patches the old Brevo/httpx code path that was replaced by `core/email.py` with Resend; (d) `test_phase12_foundation` patches `_fetch_fundamentals_stub` which was renamed/removed in a refactor. Fix: add `PortfolioDailyValue` ORM model (or raw SQL DDL in conftest); update stale tests to match current implementation.

**4. Advisor entry signals for universe use day-old screener data (P2 — Data Freshness).** `refresh-universe-entry-signals-daily` runs at 02:00 BRT. `refresh-screener-universe-daily` runs at 07:00 BRT. The entry signals batch is always computed from the previous day's screener snapshot, making signals 19+ hours stale at the moment they're written. The comment acknowledges "after screener refresh at 07h-1 day" but this is a meaningful latency gap for a trading-signals feature. Fix: shift `refresh-universe-entry-signals-daily` to 08:30 BRT (after the 08:00 FII scores calculation which depends on 07:00 screener data).

**5. `advisor/screener` limit parameter has no upper bound (P2 — Resource Exhaustion).** `app/modules/advisor/router.py:230` declares `limit: int = 100` as a plain Python default, not `Query(100, le=500)`. A caller can pass `limit=999999` and the backend will attempt to fetch and serialize the entire screener universe through the advisor service scoring loop. Fix: wrap with `Query(100, ge=1, le=500)`.

## Section 3: Security

### S1 — Idempotency-Key missing from CORS allow_headers (P0)

**File:** `backend/app/main.py:70`

```python
allow_headers=["Content-Type", "Authorization", "X-Requested-With"],
```

`Idempotency-Key` is not listed. The frontend (`investiq.com.br`) calls `api.investiq.com.br` (cross-origin). Browser CORS preflight blocks the `Idempotency-Key` header. The backend receives `None` from `request.headers.get("Idempotency-Key")` and never checks the `idempotent_checkout_requests` table. The entire P1 hotfix (migration 0029 + UpgradeCTA changes) is non-functional in production.

**Fix:** Add `"Idempotency-Key"` to the `allow_headers` list.

### S2 — SECRET_KEY not rotated in production (P2)

**File:** `backend/app/core/config.py:13` + `backend/.env`

`SECRET_KEY = "change-me-in-production"` is committed as the default AND is literally set to `change-me-in-production` in the production `.env`. Although `SECRET_KEY` is not currently used for signing anything (JWT uses RS256 RSA keys), the field exists and any future code using `settings.SECRET_KEY` would use a known value.

**Fix:** Rotate to a strong random secret in production `.env`; mark as required (no default) in config.py.

### S3 — JWT refresh tokens have no revocation mechanism (P2)

**File:** `backend/app/core/security.py`

Access tokens expire in 15 minutes (RS256). Refresh tokens expire in 7 days but are pure JWTs — no server-side blacklist. A stolen refresh token is valid for 7 days even after a user changes their password or logs out.

**Fix:** Store refresh token hash in Redis/DB on issue; check existence on use; delete on logout or password change.

### S4 — Stripe webhook stores events AFTER handler error (P2)

**File:** `backend/app/modules/billing/router.py:183-196`

The `StripeEvent` idempotency record is written even when the handler raises an exception (`handler_status = "error"`). This is intentional (prevents infinite Stripe retries on broken handlers). However, a handler that fails partially (e.g., user row locked, subscription not updated) is permanently suppressed. A failed event won't be retried — requires manual intervention via admin/logs.

**Evidence:** `router.py:183` sets `handler_status = "error"` on exception but still records the event at `router.py:188`.

**Fix:** Consider not recording the event on handler failure for transient errors, or add a dead-letter queue view in admin.

### S5 — CORS wildcard-like behavior for methods (P3)

**File:** `backend/app/main.py:69`

`allow_methods=["GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"]` — all methods allowed. No routes use PUT. Minor attack surface reduction possible.

**Fix:** Remove `PUT` from `allow_methods`.

### S6 — No HTTP security headers on frontend (P1)

**File:** `frontend/next.config.ts`

No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, or `Strict-Transport-Security` headers configured. Next.js `headers()` callback is absent.

**Fix:** Add `async headers()` in `next.config.ts` with standard security header set (CSP, HSTS, X-Frame-Options: DENY).

### S7 — SQL string interpolation for tenant_id in RLS SET LOCAL (P2)

**File:** `backend/app/core/db.py:44`, `backend/app/core/db_sync.py:105`

```python
await session.execute(text(f"SET LOCAL rls.tenant_id = '{tenant_id}'"))
```

The tenant_id is validated as a UUID (line 43: `UUID(tenant_id)`) before interpolation — this effectively prevents injection. However, using f-string interpolation is a pattern risk: if UUID validation is ever skipped or bypassed, SQL injection becomes possible. The safer pattern uses parameterized SET.

**Fix:** Use `text("SET LOCAL rls.tenant_id = :tid").bindparams(tid=tenant_id)` (PostgreSQL supports this).

### S8 — Stripe webhook on error logs exception but returns 200 (P3)

**File:** `backend/app/modules/billing/router.py:183-196`

On handler exception, the webhook returns HTTP 200 (preventing Stripe retries) but the event is marked `status="error"`. This is a deliberate design choice, but means billing failures require proactive monitoring of `StripeEvent` table — there is no alerting configured.

**Fix:** Add monitoring alert on `StripeEvent.status = "error"` accumulation.

### Auth guards summary

All authenticated endpoints use `Depends(get_current_user)`. AI analysis, advisor/analyze, and imports use `Depends(get_user_plan)` for tier gating. Admin endpoints (`/admin/logs`, `/billing/admin/*`) use `_require_admin` checking `settings.ADMIN_EMAILS`. Webhook endpoint correctly uses `get_db` (no JWT auth — Stripe has no cookie). No unauthenticated data-reading endpoints identified beyond `/health` and `/health/freshness`.
